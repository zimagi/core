# Zimagi Notification Data Model Directory

## Overview

The `app/data/notification` directory contains the Django data model implementation for the Zimagi platform's notification system. This model is responsible for defining notification structures, managing notification group associations, and providing role-based access control for notification operations across the platform.

This directory plays a critical architectural role by implementing the persistence layer for the notification system, enabling the platform to handle event-driven communications, alerting, and messaging workflows. The notification model works in conjunction with the notification command system in `app/commands/notification` to provide a complete notification management solution.

The directory is used by:
- **Developers** working on notification management and messaging features
- **System administrators** managing platform notifications and alerts
- **AI models** analyzing and generating notification components

## Cross-Referencing

This directory integrates with several other parts of the Zimagi project:

- **Parent Context**: Part of the `app/data` directory which contains Django data model applications that form the persistence layer of the Zimagi platform

## Key Concepts and Patterns

### Notification Model Structure

The notification system implements a hierarchical data model based on the specifications in `app/spec/data/notification.yml`:

- **Notification**: Represents a notification configuration
  - Fields: name (primary key), config (DictionaryField), provider_type (CharField), variables (DictionaryField), created, updated
  - Relationships: groups (NotificationGroup), failure_groups (NotificationFailureGroup)
  - Access Control: View permissions for notification-auditor role, Edit permissions for notification-admin role
  - Dynamic Fields: group_names, failure_group_names for computed properties in API responses

- **NotificationGroup**: Junction model for notification-to-group relationships
  - Fields: id (primary key), notification (ForeignKey), group (ForeignKey), created, updated
  - Access Control: View/Edit permissions for notification-admin and notification-auditor roles

- **NotificationFailureGroup**: Junction model for notification-to-failure-group relationships
  - Fields: id (primary key), notification (ForeignKey), group (ForeignKey), created, updated
  - Access Control: View/Edit permissions for notification-admin and notification-auditor roles

### Model Generation Pattern

Following Zimagi's specification-driven approach:
- Model structure is defined in `app/spec/data/notification.yml`
- Automatically generated by the model system in `app/systems/models`
- Notification model based on the `name_resource` base model which uses name field as the primary key
- NotificationGroup and NotificationFailureGroup models based on the `id_resource` base model which generates UUID primary keys
- Incorporates the `provider` and `group` mixins for plugin-based functionality and access control
- Manual extensions can be added in `models.py` if needed

### Base Model Inheritance

The models extend base models defined in `app/spec/base/data.yml`:
- **Notification**: Extends the `name_resource` base model which uses name as primary key
- **NotificationGroup**: Extends the `id_resource` base model which generates UUID primary keys based on notification_id and group_id
- **NotificationFailureGroup**: Extends the `id_resource` base model which generates UUID primary keys based on notification_id and group_id
- All provide automatic created/updated timestamp fields
- All implement consistent ordering (Notification by name, junction models by id)
- All provide resource mixin functionality

### Mixin Composition

The models incorporate mixins from `app/spec/mixins/data.yml`:
- **Notification model**: Incorporates the `provider` mixin (which includes `config` mixin) and `group` mixin
- **NotificationGroup model**: Incorporates resource mixin only
- **NotificationFailureGroup model**: Incorporates resource mixin only
- Provider mixin enables plugin-based provider functionality with config and variables fields
- Group mixin provides group-based access control through many-to-many relationships
- Standardizes common fields like created/updated timestamps

### Provider Integration

The notification model supports plugin-based providers through the provider mixin:
- Provider type selection enables different notification delivery backends
- Variables field stores provider-specific configuration parameters
- Integration with the plugin system in `app/systems/plugins` for provider implementations
- Config mixin provides standardized configuration field management

### Access Control

The notification models implement role-based access control as defined in `app/spec/data/notification.yml`:
- **Notification model**: View permissions for notification-auditor role, Edit permissions for notification-admin role
- **NotificationGroup model**: View/Edit permissions for notification-admin and notification-auditor roles
- **NotificationFailureGroup model**: View/Edit permissions for notification-admin and notification-auditor roles

### Dynamic Fields

The notification model implements dynamic fields for computed properties:
- **group_names**: Provides a formatted display of all groups associated with a notification
- **failure_group_names**: Provides a formatted display of all failure groups associated with a notification

### Naming Conventions

- Model files are consistently named `models.py`
- Migration files follow Django's sequential naming pattern
- Class names use PascalCase and are descriptive of their domain (Notification, NotificationGroup, NotificationFailureGroup)
- Field names use snake_case following Django conventions
- Database tables use prefixed names (core_notification, core_notification_group, core_notification_failure_group)

### File Organization

Files are organized by Django application conventions:
- Model definition in `models.py`
- Database migrations in the `migrations` subdirectory
- One migration file per schema change

### Domain-Specific Patterns

- The models extend base models from `app/spec/base/data.yml` for consistent resource behavior
- Uses facade pattern from `app/systems/models` for business logic implementation
- Integrates with Django's migration system for schema version control
- Implements the `provider` and `group` mixins for plugin-based functionality and access control
- Notification model uses the `name_resource` base which uses name as the primary key
- NotificationGroup and NotificationFailureGroup models use the `id_resource` base which generates a UUID primary key based on specified id_fields
- Implements ordering by name for Notification model and by id for junction models
- Implements dynamic fields for computed properties in API responses

## Developer Notes and Usage Tips

### Integration Requirements

This model requires:
- Django framework access for ORM operations
- Proper database configuration in settings
- Integration with the group data model in `app/data/group` for access control
- Specification files in `app/spec/data/notification.yml` and `app/spec/base/data.yml` for model generation
- Plugin system access for provider functionality

### Usage Patterns

- Models are accessed through facade classes rather than directly
- Business logic should be implemented in facade methods
- New models should be defined in specifications first
- Migration files should be generated using Django's migration system
- Dynamic fields like group_names should be implemented in facade classes

### Dependencies

- Django ORM for database interactions
- Model systems from `app/systems/models` for dynamic generation
- Specification files from `app/spec/data/notification.yml` and `app/spec/base/data.yml` for model definitions
- Group model from `app/data/group` for access control relationships
- Plugin system from `app/systems/plugins` for provider functionality
