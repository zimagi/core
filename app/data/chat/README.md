# Zimagi Chat Data Models Directory

## Overview

The `app/data/chat` directory contains Django data model implementations for the Zimagi platform's chat and messaging functionality. This directory defines the core data structures that enable conversational interfaces, message tracking, and dialog management within the platform.

This directory plays a critical architectural role by implementing the persistence layer for chat-related features, providing the foundation for AI interactions, user messaging, and conversational workflows. The models here are automatically generated from specifications in `app/spec/data/chat.yml` through the meta-programming system in `app/systems/models`.

The directory is used by:
- **Developers** working on chat functionality and messaging systems
- **AI models** analyzing and generating conversational components
- **Build systems** that process specifications into database migrations
- **System administrators** managing data schemas and relationships

## Key Concepts and Patterns

### Chat Model Structure

The chat system implements a hierarchical data model based on the specifications in `app/spec/data/chat.yml`:

- **Chat**: Represents a conversation context, associated with a specific user
  - Fields: id (primary key), name, user (foreign key), label, created, updated
  - Relationships: dialogs (ChatDialog), messages (ChatMessage)
  - Access Control: View/Edit permissions for chat-user role

- **ChatDialog**: Represents a conversation thread or context within a chat
  - Fields: id (primary key), chat (foreign key), created, updated
  - Relationships: messages (ChatMessage)
  - Access Control: View permissions for chat-user role, no edit permissions

- **ChatMessage**: Individual messages within dialogs, with role and content information
  - Fields: id (primary key), chat (foreign key), dialog (foreign key), role, sender, content, created, updated
  - Access Control: View permissions for chat-user role, no edit permissions

### Model Generation Pattern

Following Zimagi's specification-driven approach:
- Model structure is defined in `app/spec/data/chat.yml`
- Automatically generated by the model system in `app/systems/models`
- Based on the `id_resource` base model from `app/spec/base/data.yml` which provides UUID-based identification
- Manual extensions can be added in `models.py` if needed

### Base Model Inheritance

All chat models inherit from the `id_resource` base model defined in `app/spec/base/data.yml`:
- Provides UUID primary key generation based on name field
- Includes automatic created/updated timestamp fields
- Implements consistent ordering by name field
- Provides resource mixin functionality

### Chat Message Roles

The ChatMessage model implements role-based messaging:
- **user**: Messages sent by human users
- **assistant**: Messages sent by AI assistants or system responses

### Access Control

Chat models implement role-based access control as defined in `app/spec/data/chat.yml`:
- Chat model: View/Edit permissions for chat-user role
- ChatDialog model: View permissions for chat-user role, no edit permissions
- ChatMessage model: View permissions for chat-user role, no edit permissions

### Naming Conventions

- Model files are consistently named `models.py`
- Migration files follow Django's sequential naming pattern
- Class names use PascalCase and are descriptive of their domain (Chat, ChatMessage, ChatDialog)
- Field names use snake_case following Django conventions
- Database tables use prefixed names (core_chat, core_chat_dialog, core_chat_message)

### File Organization

Files are organized by Django application conventions:
- Model definition in `models.py`
- Database migrations in the `migrations` subdirectory
- One migration file per schema change

### Domain-Specific Patterns

- The model extends the `id_resource` base model from `app/spec/base/data.yml` for consistent resource behavior
- Uses facade pattern from `app/systems/models` for business logic implementation
- Integrates with Django's migration system for schema version control
- Implements ordering by creation time for chronological message display
- Uses the `id_resource` base which generates a UUID primary key based on specified id_fields
- Implements dynamic fields for computed properties in API responses

## Developer Notes and Usage Tips

### Integration Requirements

These models require:
- Django framework access for ORM operations
- Proper database configuration in settings
- Integration with the user data model in `app/data/user`
- Specification files in `app/spec/data/chat.yml` and `app/spec/base/data.yml` for model generation

### Usage Patterns

- Models are accessed through facade classes rather than directly
- Business logic should be implemented in facade methods
- New models should be defined in specifications first
- Migrations should be generated using Django's migration system
- Dynamic fields like message_render should be implemented in facade classes

### Dependencies

- Django ORM for database interactions
- User model from `app/data/user` for relationship management
- Model systems from `app/systems/models` for dynamic generation
- Specification files from `app/spec/data/chat.yml` and `app/spec/base/data.yml` for model definitions

### AI Development Guidance

When generating or modifying chat model components:

1. Maintain consistency with specification-driven generation patterns
2. Follow established naming conventions for models and fields
3. Ensure proper relationships and constraints are defined for data integrity
4. Consider performance implications for chat query operations that may run frequently
5. Maintain consistency with existing API and command integration patterns
6. Document all public methods with clear docstrings
7. Follow the established patterns for facade and base class usage
8. Respect the role-based access control configuration defined in the specification
9. Ensure proper id_fields configuration for UUID generation
10. Implement dynamic fields in facade classes for computed properties
