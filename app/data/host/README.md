# Zimagi Host Data Model Directory

## Overview

The `app/data/host` directory contains the Django data model implementation for the Zimagi platform's host connection management functionality. This model is responsible for defining host configurations, managing remote host credentials, and providing programmatic access to remote Zimagi instances through command and data APIs.

This directory plays a critical architectural role by implementing the persistence layer for host management, enabling the platform to maintain connections to remote Zimagi services and interact with them programmatically. The host model works in conjunction with the host command system in `app/commands/host` to provide a complete remote host management solution.

The directory is used by:
- **Developers** working on remote host management and service integration
- **System administrators** managing connections to remote Zimagi instances
- **AI models** analyzing and generating host connection components

## Key Concepts and Patterns

### Host Model Structure

The Host model implements a comprehensive host connection management mechanism based on the specifications in `app/spec/data/host.yml`:
- **name**: Primary key identifier for the host entry (CharField, max_length: 256)
- **host**: URL of the remote host (URLField)
- **command_port**: Port number for command API access (IntegerField, null: True)
- **data_port**: Port number for data API access (IntegerField, null: True)
- **user**: Username for remote authentication (CharField, max_length: 150, null: True)
- **token**: Authentication token for remote access (CharField, max_length: 256, null: True)
- **encryption_key**: Key used for encrypting communications with the remote host (CharField, max_length: 256, null: True)
- **created/updated**: Automatic timestamp fields for tracking entry lifecycle

### Model Generation Pattern

Following Zimagi's specification-driven approach:
- Model structure is defined in `app/spec/data/host.yml`
- Automatically generated by the model system in `app/systems/models`
- Based on the `name_resource` base model which uses the name field as the primary key
- Incorporates the `resource` mixin for standardized functionality
- Manual extensions can be added in `models.py` if needed

### API Client Integration

The host model provides direct integration with Zimagi's client libraries:
- **command_api()**: Returns a CommandClient instance for executing remote commands
- **data_api()**: Returns a DataClient instance for accessing remote data APIs
- Both methods respect port availability and return None if the respective port is not configured
- Client instances are pre-configured with host credentials for seamless remote access

### Access Control

The host model implements role-based access control as defined in `app/spec/data/host.yml`:
- **View permissions**: Available to users with host-admin and host-user roles
- **Edit permissions**: Restricted to users with host-admin roles
- **Destroy operations**: Uses default behavior with proper access control

### Base Model Inheritance

The model extends the `name_resource` base model defined in `app/spec/base/data.yml`:
- Uses name as primary key for direct identification
- Provides automatic created/updated timestamp fields
- Implements consistent ordering by name field
- Provides resource mixin functionality

### Mixin Composition

The model incorporates the resource mixin from `app/spec/mixins/data.yml`:
- **ResourceMixin**: Provides core resource functionality and metadata management
- Standardizes common fields like created/updated timestamps
- Ensures consistent behavior across all resource-based models

### Naming Conventions

- Model files are consistently named `models.py`
- Migration files follow Django's sequential naming pattern
- Class names use PascalCase and are descriptive of their domain (Host, HostFacade)
- Field names use snake_case following Django conventions
- Database table uses prefixed name (core_host)
- Method names are descriptive and indicate their purpose (command_api, data_api)

### File Organization

Files are organized by Django application conventions:
- Model definition in `models.py`
- Database migrations in the `migrations` subdirectory
- One migration file per schema change

### Domain-Specific Patterns

- The model extends the `name_resource` base model from `app/spec/base/data.yml` for consistent resource behavior
- Uses facade pattern from `app/systems/models` for business logic implementation
- Integrates with Django's migration system for schema version control
- Implements the `resource` mixin for standardized functionality
- Uses the `name_resource` base which uses name as the primary key
- Implements ordering by name for consistent host listing
- Provides direct integration with client libraries for remote API access
