# Zimagi Scaling Event Data Model Directory

## Overview

The `app/data/scaling_event` directory contains the Django data model implementation for the Zimagi platform's auto-scaling event tracking functionality. This model is responsible for recording and managing scaling activities that occur within the platform's worker management system, providing insights into resource allocation and system performance during scaling operations.

This directory plays a critical architectural role by implementing the persistence layer for scaling event tracking, enabling the platform to maintain historical records of worker scaling activities for monitoring, analysis, and optimization purposes. The scaling event model works in conjunction with the scaling command system in `app/commands/scaling` to provide a complete auto-scaling management solution.

The directory is used by:
- **Developers** working on auto-scaling and resource management features
- **System administrators** monitoring platform performance and resource utilization
- **AI models** analyzing scaling patterns and generating resource optimization components

## Key Concepts and Patterns

### Scaling Event Model Structure

The ScalingEvent model implements a comprehensive scaling event tracking mechanism based on the specifications in `app/spec/data/scaling.yml`:
- **name**: Primary key identifier for the scaling event entry (CharField, max_length: 256)
- **command**: The command that triggered the scaling event (CharField, max_length: 255)
- **worker_type**: The type of worker being scaled (CharField, max_length: 255)
- **worker_max_count**: Maximum number of workers allowed (IntegerField, null: True)
- **worker_count**: Current number of workers (IntegerField, null: True)
- **task_count**: Number of tasks in the queue (IntegerField, null: True)
- **workers_created**: Number of workers created during the scaling event (IntegerField, null: True)
- **created/updated**: Automatic timestamp fields for tracking entry lifecycle

### Model Generation Pattern

Following Zimagi's specification-driven approach:
- Model structure is defined in `app/spec/data/scaling.yml`
- Automatically generated by the model system in `app/systems/models`
- Based on the `name_resource` base model which uses the name field as the primary key
- Manual extensions can be added in `models.py` if needed

### Access Control

The scaling event model implements role-based access control as defined in `app/spec/data/scaling.yml`:
- **View permissions**: Available to users with admin roles
- **Edit permissions**: No roles have edit permissions (model is read-only for updates)
- **Destroy operations**: Disabled entirely (destroy operations are disabled)

### Base Model Inheritance

The model extends the `name_resource` base model defined in `app/spec/base/data.yml`:
- Uses name as primary key for direct identification
- Provides automatic created/updated timestamp fields
- Implements consistent ordering by creation time (descending)
- Provides resource mixin functionality

### Naming Conventions

- Model files are consistently named `models.py`
- Migration files follow Django's sequential naming pattern
- Class names use PascalCase and are descriptive of their domain (ScalingEvent)
- Field names use snake_case following Django conventions
- Database table uses prefixed name (core_scaling_event)

### File Organization

Files are organized by Django application conventions:
- Model definition in `models.py`
- Database migrations in the `migrations` subdirectory
- One migration file per schema change

### Domain-Specific Patterns

- The model extends the `name_resource` base model from `app/spec/base/data.yml` for consistent resource behavior
- Uses facade pattern from `app/systems/models` for business logic implementation
- Integrates with Django's migration system for schema version control
- Uses the `name_resource` base which uses name as the primary key
- Implements ordering by creation time (descending) for chronological event display
- Uses automatic name generation in the save method for unique identification
