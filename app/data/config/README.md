# Zimagi Configuration Data Model Directory

## Overview

The `app/data/config` directory contains the Django data model implementation for the Zimagi platform's configuration management system. This model is responsible for storing system configurations, managing configuration values with type safety, and providing role-based access control for configuration data.

This directory plays a critical architectural role by implementing the persistence layer for system configurations, enabling the platform to maintain runtime settings, module configurations, and user-defined variables. The configuration model works in conjunction with the configuration command system to provide a complete configuration management solution.

The directory is used by:
- **Developers** working on system configuration and settings management
- **System administrators** managing platform configurations
- **AI models** analyzing and generating configuration components

This directory integrates with several other parts of the Zimagi project:

## Key Concepts and Patterns

### Config Model Structure

The Config model implements a flexible configuration storage mechanism based on the specifications:
- **name**: Primary key identifier for the configuration entry (CharField, max_length: 256)
- **config**: Dictionary field storing configuration metadata (DictionaryField, default: dict)
- **provider_type**: Specifies the configuration provider implementation (CharField, max_length: 128, default: "base")
- **variables**: Dictionary field storing provider-specific variables (DictionaryField, default: dict, not editable)
- **value**: The actual configuration value stored with flexible data typing (DataField, null: True)
- **value_type**: Specifies the data type of the configuration value (CharField, max_length: 150, default: "str")
- **groups**: Many-to-many relationship for role-based access control (ManyToManyField to Group model)
- **created/updated**: Automatic timestamp fields for tracking entry lifecycle

### Model Generation Pattern

Following Zimagi's specification-driven approach:
- Model structure is defined in `app/spec/data/config.yml`
- Automatically generated by the model system in `app/systems/models`
- Based on the `name_resource` base model which uses the name field as the primary key
- Incorporates the `provider` and `group` mixins for plugin-based functionality and access control
- Manual extensions can be added in `models.py` if needed

### Provider Integration

The configuration model supports plugin-based providers through the provider mixin:
- Provider type selection enables different configuration storage backends
- Variables field stores provider-specific configuration parameters
- Integration with the plugin system for provider implementations
- Config mixin provides standardized configuration field management

### Access Control

The config model implements role-based access control as defined in `app/spec/data/config.yml`:
- **View permissions**: Restricted to config-auditor role
- **Edit permissions**: Restricted to config-admin role
- **Destroy operations**: Uses default behavior with proper access control

### Base Model Inheritance

The model extends the `name_resource` base model defined in `app/spec/base/data.yml`:
- Uses name as primary key for direct identification
- Provides automatic created/updated timestamp fields
- Implements consistent ordering by name field
- Provides resource mixin functionality

### Mixin Composition

The model incorporates multiple mixins from `app/spec/mixins/data.yml`:
- **ProviderMixin**: Enables plugin-based provider functionality with config and variables fields
- **GroupMixin**: Provides group-based access control through many-to-many relationships
- **ResourceMixin**: Provides core resource functionality and metadata management

### Naming Conventions

- Model files are consistently named `models.py`
- Migration files follow Django's sequential naming pattern
- Class names use PascalCase and are descriptive of their domain (Config)
- Field names use snake_case following Django conventions
- Database table uses prefixed name (core_config)

### File Organization

Files are organized by Django application conventions:
- Model definition in `models.py`
- Database migrations in the `migrations` subdirectory
- One migration file per schema change

### Domain-Specific Patterns

- The model extends the `name_resource` base model from `app/spec/base/data.yml` for consistent resource behavior
- Uses facade pattern for business logic implementation
- Integrates with Django's migration system for schema version control
- Implements the `provider` and `group` mixins for plugin-based functionality and access control
- Uses the `name_resource` base which uses name as primary key
- Implements ordering by name for consistent configuration listing

## Developer Notes and Usage Tips

### Integration Requirements

This model requires:
- Django framework access for ORM operations
- Proper database configuration in settings
- Integration with the group data model in `app/data/group` for access control
- Specification file in `app/spec/data/config.yml` for model generation

- Provider implementations should extend the base provider

- Model systems from `app/systems/models` for dynamic generation
- Specification file from `app/spec/data/config.yml` for model definition
- Group model from `app/data/group` for access control relationships
- Plugin system for provider functionality
