# Zimagi State Data Model Directory

## Overview

The `app/data/state` directory contains the Django data model implementation for the Zimagi platform's runtime state management functionality. This model is responsible for storing and managing runtime state variables that track the operational status and configuration of various platform components.

This directory plays a critical architectural role by implementing the persistence layer for state management, enabling the platform to maintain runtime configuration values, execution states, and system status information. The state model works in conjunction with the state command system in `app/commands/state` to provide a complete state management solution.

The directory is used by:
- **Developers** working on system state management and runtime configuration
- **System administrators** monitoring and managing platform state
- **AI models** analyzing and generating state management components

## Cross-Referencing

This directory integrates with several other parts of the Zimagi project:

- **Parent Context**: Part of the `app/data` directory which contains Django data model applications that form the persistence layer of the Zimagi platform
- **Specifications**: The state data model is defined in `app/spec/data/state.yml` which drives the automatic generation of the model
- **Base Models**: Extends the `name_resource` base model defined in `app/spec/base/data.yml`
- **API Systems**: Connects to `app/systems/api/data` for REST API exposure of state variables
- **Command Systems**: Works with `app/commands/state` for state command implementations
- **Settings**: Integrates with configurations defined in `app/settings` for state management parameters

## Key Concepts and Patterns

### State Model Structure

The State model implements a simple state management mechanism based on the specifications in `app/spec/data/state.yml`:
- **name**: Primary key identifier for the state entry (CharField, max_length: 256)
- **value**: The actual state value stored with flexible data typing (DataField, null: True)
- **created/updated**: Automatic timestamp fields for tracking entry lifecycle

### Model Generation Pattern

Following Zimagi's specification-driven approach:
- Model structure is defined in `app/spec/data/state.yml`
- Automatically generated by the model system in `app/systems/models`
- Based on the `name_resource` base model which uses the name field as the primary key
- Manual extensions can be added in `models.py` if needed

### Access Control

The state model implements role-based access control as defined in `app/spec/data/state.yml`:
- **View permissions**: Available to users with config-auditor roles
- **Edit permissions**: Restricted to users with config-admin roles
- **Destroy operations**: Uses default behavior with proper access control

### Base Model Inheritance

The model extends the `name_resource` base model defined in `app/spec/base/data.yml`:
- Uses name as primary key for direct identification
- Provides automatic created/updated timestamp fields
- Implements consistent ordering by name field
- Provides resource mixin functionality

### Naming Conventions

- Model files are consistently named `models.py`
- Migration files follow Django's sequential naming pattern
- Class names use PascalCase and are descriptive of their domain (State)
- Field names use snake_case following Django conventions
- Database table uses prefixed name (core_state)

### File Organization

Files are organized by Django application conventions:
- Model definition in `models.py`
- Database migrations in the `migrations` subdirectory
- One migration file per schema change

### Domain-Specific Patterns

- The model extends the `name_resource` base model from `app/spec/base/data.yml` for consistent resource behavior
- Uses facade pattern from `app/systems/models` for business logic implementation
- Integrates with Django's migration system for schema version control
- Uses the `name_resource` base which uses name as primary key
- Implements ordering by name for consistent state listing

## Developer Notes and Usage Tips

### Integration Requirements

This model requires:
- Django framework access for ORM operations
- Proper database configuration in settings
- Specification file in `app/spec/data/state.yml` for model generation

- The model is automatically used by state commands in `app/commands/state`
- Access through the facade rather than direct model interaction
- Business logic should be implemented in facade methods if extensions are needed

### Dependencies

- Django ORM for database interactions
- Model systems from `app/systems/models` for dynamic generation
- Specification file from `app/spec/data/state.yml` for model definition
