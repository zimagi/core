# Zimagi Log Data Model Directory

## Overview

The `app/data/log` directory contains the Django data model implementation for the Zimagi platform's logging and audit functionality. This model is responsible for tracking command execution, system events, and operational activities across the platform, providing insights into system behavior and enabling audit trails for security and compliance purposes.

This directory plays a critical architectural role by implementing the persistence layer for system logging, enabling the platform to maintain detailed records of operations, errors, and user activities. The log model works in conjunction with the logging system in `app/systems/commands/messages.py` to provide a complete logging solution.

The directory is used by:
- **Developers** working on system monitoring and debugging features
- **System administrators** managing platform operations and security
- **AI models** analyzing platform behavior and performance patterns

## Cross-Referencing

This directory integrates with several other parts of the Zimagi project:

- **User System**: Integrates with `app/data/user` for user authentication and relationship management
- **Schedule System**: Integrates with `app/data/schedule` for scheduled task relationship management

## Key Concepts and Patterns

### Log Model Structure

The log system implements a hierarchical data model based on the specifications in `app/spec/data/log.yml`:

- **Log**: Represents a command execution or system operation context
  - Fields: name (primary key), config (DictionaryField), command (CharField), status (CharField), task_id (CharField), worker (CharField), user (ForeignKey to User), schedule (ForeignKey to ScheduledTask), created, updated
  - Relationships: messages (LogMessage)
  - Status options: running, success, failed
  - Access Control: View permissions for user-auditor role, no edit permissions
  - Operations: Update and destroy operations are disabled

- **LogMessage**: Individual messages or events within a log context
  - Fields: id (primary key), data (DataField), log (ForeignKey to Log), created, updated
  - Data field contains serialized message information from command execution
  - Access Control: View permissions for user-auditor role, no edit permissions
  - API Exposure: Not exposed through API endpoints

### Model Generation Pattern

Following Zimagi's specification-driven approach:
- Model structure is defined in `app/spec/data/log.yml`
- Automatically generated by the model system in `app/systems/models`
- Based on the `name_resource` base model which uses the name field as the primary key for Log model
- LogMessage model is based on the `id_resource` base model which generates UUID primary keys
- Incorporates the `config` mixin for configuration storage
- Manual extensions can be added in `models.py` if needed

### Base Model Inheritance

The models extend base models defined in `app/spec/base/data.yml`:
- **Log**: Extends the `name_resource` base model which uses name as primary key
- **LogMessage**: Extends the `id_resource` base model which generates UUID primary keys based on specified id_fields
- Both provide automatic created/updated timestamp fields
- Both implement consistent ordering (Log by creation time descending, LogMessage by creation time descending)
- Both provide resource mixin functionality

### Mixin Composition

The models incorporate mixins from `app/spec/mixins/data.yml`:
- **Log model**: Incorporates the `config` mixin which provides configuration storage through a DictionaryField
- **LogMessage model**: Does not incorporate any mixins
- Standardizes common fields like created/updated timestamps
- Ensures consistent behavior across all resource-based models

### Access Control

The log models implement role-based access control as defined in `app/spec/data/log.yml`:
- **Log model**: View permissions for user-auditor role, no edit permissions
- **LogMessage model**: View permissions for user-auditor role, no edit permissions
- **Destroy operations**: Disabled entirely for Log model, not applicable for LogMessage model

### Dynamic Fields

The log model implements dynamic fields for computed properties:
- **message_render**: Provides a formatted display of all log messages associated with a log entry

### Naming Conventions

- Model files are consistently named `models.py`
- Migration files follow Django's sequential naming pattern
- Class names use PascalCase and are descriptive of their domain (Log, LogMessage, LogFacade)
- Field names use snake_case following Django conventions
- Database tables use prefixed names (core_log, core_log_message)

### File Organization

Files are organized by Django application conventions:
- Model definition in `models.py`
- Database migrations in the `migrations` subdirectory
- One migration file per schema change

### Domain-Specific Patterns

- The models extend base models from `app/spec/base/data.yml` for consistent resource behavior
- Uses facade pattern from `app/systems/models` for business logic implementation
- Integrates with Django's migration system for schema version control
- Implements the `config` mixin for configuration storage in Log model
- Log model uses the `name_resource` base which uses name as the primary key
- LogMessage model uses the `id_resource` base which generates a UUID primary key based on log_id and created fields
- Implements ordering by creation time (descending) for chronological log display
- Implements dynamic fields for computed properties in API responses
- Supports integration with command execution through message serialization in LogMessage data field

## Developer Notes and Usage Tips

### Integration Requirements

This model requires:
- Django framework access for ORM operations
- Proper database configuration in settings
- Integration with the user data model in `app/data/user`
- Integration with the schedule data model in `app/data/schedule`
- Specification files in `app/spec/data/log.yml` for model generation

### Usage Patterns

- Models are accessed through facade classes rather than directly
- Business logic should be implemented in facade methods
- New models should be defined in specifications first
- Migration files should be generated using Django's migration system when changes are made
- The message_render dynamic field provides formatted display of log messages
- Log messages are serialized using the AppMessage system from `app/systems/commands/messages.py`

### Dependencies

- Django ORM for database interactions
- User model from `app/data/user` for relationship management
- Schedule model from `app/data/schedule` for relationship management
- Model systems from `app/systems/models` for dynamic generation
- Specification files from `app/spec/data/log.yml` and `app/spec/base/data.yml` for model definitions
- Command message system from `app/systems/commands/messages.py` for message serialization
