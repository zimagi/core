# Zimagi Schedule Data Model Directory

## Overview

The `app/data/schedule` directory contains the Django data model implementation for the Zimagi platform's task scheduling functionality. This model is responsible for defining scheduled task structures, managing task intervals, crontabs, and datetime-based schedules, and providing role-based access control for scheduled operations across the platform.

This directory plays a critical architectural role by implementing the persistence layer for scheduled tasks, enabling the platform to handle recurring and time-based operations through integration with the Celery distributed task queue. The schedule models work in conjunction with the Celery system in `app/systems/celery` to provide a complete task scheduling solution.

The directory is used by:
- **Developers** working on task scheduling and automation features
- **System administrators** managing platform scheduled tasks
- **AI models** analyzing and generating scheduling components

## Key Concepts and Patterns

### Schedule Model Structure

The schedule models implement a comprehensive task scheduling mechanism based on the specifications in `app/spec/data/schedule.yml`:

- **ScheduledTask**: Represents a scheduled task that can be executed by the Celery worker system
  - Fields: name (primary key), args (ListField), kwargs (DictionaryField), headers (DictionaryField)
  - Relationships: user (ForeignKey to User), interval (ForeignKey to TaskInterval), crontab (ForeignKey to TaskCrontab), clocked (ForeignKey to TaskDatetime)
  - Access Control: View permissions for schedule-auditor role, Edit permissions for schedule-admin role
  - Command Base: schedule
  - Ordering: by created field

- **TaskInterval**: Represents an interval-based schedule (e.g., every 5 minutes)
  - Base Model: id_resource (inherits from IdentifierResourceBase)
  - Access Control: View permissions for schedule-auditor role, Edit permissions for schedule-admin role

- **TaskCrontab**: Represents a cron-based schedule (e.g., at 2:30 AM every Monday)
  - Base Model: id_resource (inherits from IdentifierResourceBase)
  - Access Control: View permissions for schedule-auditor role, Edit permissions for schedule-admin role

- **TaskDatetime**: Represents a one-time scheduled execution at a specific datetime
  - Base Model: id_resource (inherits from IdentifierResourceBase)
  - Access Control: View permissions for schedule-auditor role, Edit permissions for schedule-admin role

- **ScheduledTaskChanges**: Tracks changes to scheduled tasks for synchronization purposes
  - Fields: ident (primary key), last_update (DateTimeField)
  - Access Control: System-level model with restricted access
  - Purpose: Maintains last update timestamps for cache invalidation

### Model Generation Pattern

Following Zimagi's specification-driven approach:
- Model structures are defined in `app/spec/data/schedule.yml`
- Automatically generated by the model system in `app/systems/models`
- Based on the `name_resource` base model for ScheduledTask and `id_resource` base model for schedule type models
- Manual extensions can be added in `models.py` if needed

### Base Model Inheritance

The models extend base models defined in `app/spec/base/data.yml`:
- ScheduledTask extends `name_resource` base model which uses name as primary key
- TaskInterval, TaskCrontab, and TaskDatetime extend `id_resource` base model which generates UUID-based identifiers
- ScheduledTaskChanges is a direct extension of django_celery_beat models
- All models provide automatic created/updated timestamp fields
- All models implement consistent ordering by name field

### Celery Integration

The schedule models integrate directly with Celery's periodic task system:
- Leverages django-celery-beat for database-backed scheduling
- Supports interval-based scheduling through TaskInterval model
- Supports cron-based scheduling through TaskCrontab model
- Supports one-time scheduled execution through TaskDatetime model
- Tracks task execution history and status
- Uses ScheduledTaskChanges for tracking modification timestamps

### Access Control

The schedule models implement role-based access control as defined in `app/spec/data/schedule.yml`:
- **View permissions**: Available to users with schedule-auditor roles
- **Edit permissions**: Restricted to users with schedule-admin roles
- **Destroy operations**: Uses default behavior with proper access control

### Naming Conventions

- Model files are consistently named `models.py`
- Migration files follow Django's sequential naming pattern
- Class names use PascalCase and are descriptive of their domain (ScheduledTask, TaskInterval, TaskCrontab, TaskDatetime, ScheduledTaskChanges)
- Field names use snake_case following Django conventions
- Database tables use prefixed names (core_scheduled_task, core_task_interval, core_task_crontab, core_task_datetime, core_task_changes)

### File Organization

Files are organized by Django application conventions:
- Model definition in `models.py`
- Database migrations in the `migrations` subdirectory
- One migration file per schema change

### Domain-Specific Patterns

- The models extend the base models from `app/spec/base/data.yml` for consistent resource behavior
- Uses facade pattern from `app/systems/models` for business logic implementation
- Integrates with Django's migration system for schema version control
- Implements ordering by creation time for consistent schedule listing
- Integrates with django-celery-beat for Celery task scheduling
- Uses custom field types (@zimagi.ListField, @zimagi.DictionaryField) for specialized data storage
- Implements change tracking through ScheduledTaskChanges for cache invalidation
