# Zimagi Module Data Model Directory

## Overview

The `app/data/module` directory contains the Django data model implementation for the Zimagi platform's module management functionality. This model is responsible for defining module structures, managing remote module configurations, and providing role-based access control for module operations across the platform.

This directory plays a critical architectural role by implementing the persistence layer for the module system, enabling the platform to handle extensible functionality through modular components. The module model works in conjunction with the module command system in `app/commands/module` to provide a complete module management solution.

The directory is used by:
- **Developers** working on module management and extensibility features
- **System administrators** managing platform modules and extensions
- **AI models** analyzing and generating module components

This directory integrates with several other parts of the Zimagi project:

## Key Concepts and Patterns

### Module Model Structure

The Module model implements a comprehensive module management mechanism based on the specifications in `app/spec/data/module.yml`:
- **name**: Primary key identifier for the module entry (CharField, max_length: 256)
- **config**: Dictionary field storing module metadata (DictionaryField, default: dict)
- **provider_type**: Specifies the module provider implementation (CharField, max_length: 128, default: "base")
- **variables**: Dictionary field storing provider-specific variables (DictionaryField, default: dict, not editable)
- **remote**: URL or path to remote module source (CharField, max_length: 256, null: True)
- **reference**: Version reference for the module (CharField, max_length: 128, null: True)
- **groups**: Many-to-many relationship for role-based access control (ManyToManyField to Group model)
- **created/updated**: Automatic timestamp fields for tracking entry lifecycle

### Model Generation Pattern

Following Zimagi's specification-driven approach:
- Model structure is defined in `app/spec/data/module.yml`
- Automatically generated by the model system in `app/systems/models`
- Based on the `name_resource` base model which uses the name field as the primary key
- Incorporates the `provider` and `group` mixins for plugin-based functionality and access control
- Manual extensions can be added in `models.py` if needed

### Provider Integration

The module model supports plugin-based providers through the provider mixin:
- Provider type selection enables different module storage and retrieval backends
- Variables field stores provider-specific configuration parameters
- Integration with the plugin system in `app/systems/plugins` for provider implementations
- Config mixin provides standardized configuration field management

### Access Control

The module model implements role-based access control as defined in `app/spec/data/module.yml`:
- **View permissions**: Available to users with module-auditor roles
- **Edit permissions**: Restricted to users with module-admin roles
- **Destroy operations**: Uses default behavior with proper access control

### Base Model Inheritance

The model extends the `name_resource` base model defined in `app/spec/base/data.yml`:
- Uses name as primary key for direct identification
- Provides automatic created/updated timestamp fields
- Implements consistent ordering by provider type and name fields
- Provides resource mixin functionality

### Mixin Composition

The model incorporates multiple mixins from `app/spec/mixins/data.yml`:
- **ProviderMixin**: Enables plugin-based provider functionality with config and variables fields
- **GroupMixin**: Provides group-based access control through many-to-many relationships
- **ResourceMixin**: Provides core resource functionality and metadata management

### Naming Conventions

- Model files are consistently named `models.py`
- Migration files follow Django's sequential naming pattern
- Class names use PascalCase and are descriptive of their domain (Module, ModuleFacade)
- Field names use snake_case following Django conventions
- Database table uses prefixed name (core_module)

### File Organization

Files are organized by Django application conventions:
- Model definition in `models.py`
- Database migrations in the `migrations` subdirectory
- One migration file per schema change

### Domain-Specific Patterns

- The model extends the `name_resource` base model from `app/spec/base/data.yml` for consistent resource behavior
- Uses facade pattern from `app/systems/models` for business logic implementation
- Integrates with Django's migration system for schema version control
- Implements the `provider` and `group` mixins for plugin-based functionality and access control
- Uses the `name_resource` base which uses name as the primary key
- Implements ordering by provider type (descending) and name for consistent module listing
- Implements module status tracking with validation and dirty checking
- Supports module version and compatibility information
- Provides deploy module configuration saving functionality

## Developer Notes and Usage Tips

### Integration Requirements

This model requires:
- Django framework access for ORM operations
- Proper database configuration in settings
- Integration with the group data model in `app/data/group` for access control
- Specification file in `app/spec/data/module.yml` for model generation
- Plugin system access for provider functionality

### Usage Patterns

- The model is automatically used by module commands in `app/commands/module`
- Access through the facade rather than direct model interaction
- Business logic should be implemented in facade methods if extensions are needed
- Provider implementations should extend the base provider in `app/plugins/module`
- Migration files should be generated using Django's migration system when changes are made

### Dependencies

- Django ORM for database interactions
- Model systems from `app/systems/models` for dynamic generation
- Specification file from `app/spec/data/module.yml` for model definition
- Group model from `app/data/group` for access control relationships
- Plugin system from `app/systems/plugins` for provider functionality
- Utility functions from `app/utility` for filesystem and serialization operations
