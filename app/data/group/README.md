# Zimagi Group Data Model Directory

## Overview

The `app/data/group` directory contains the Django data model implementation for the Zimagi platform's user group and role-based access control system. This model is responsible for defining group structures, managing hierarchical relationships between groups, and providing the foundation for role-based permissions across the platform.

This directory plays a critical architectural role by implementing the persistence layer for access control, enabling the platform to manage user permissions, organize resources by group membership, and enforce security policies through group-based access control. The group model works in conjunction with group-related commands to provide a complete access management solution.

The directory is used by:
- **Developers** working on access control and permission management features
- **System administrators** managing user groups and permissions
- **AI models** analyzing and generating access control components

## Key Concepts and Patterns

### Group Model Structure

The Group model implements a hierarchical group management mechanism based on the specifications in `app/spec/data/group.yml`:
- **name**: Primary key identifier for the group entry (CharField, max_length: 256)
- **config**: Dictionary field storing group metadata (DictionaryField, default: dict)
- **provider_type**: Specifies the group provider implementation (CharField, max_length: 128, default: "base")
- **variables**: Dictionary field storing provider-specific variables (DictionaryField, default: dict, not editable)
- **parent**: Self-referential foreign key for hierarchical group relationships (ForeignKey to Group model, null: True)
- **groups**: Many-to-many relationship field for group associations (ManyToManyField to Group model, blank: true)
- **created/updated**: Automatic timestamp fields for tracking entry lifecycle

### Model Generation Pattern

Following Zimagi's specification-driven approach:
- Model structure is defined in `app/spec/data/group.yml`
- Automatically generated by the model system in `app/systems/models`
- Based on the `name_resource` base model which uses the name field as the primary key
- Incorporates the `provider` mixin for plugin-based functionality
- Manual extensions can be added in `models.py` if needed

### Provider Integration

The group model supports plugin-based providers through the provider mixin:
- Provider type selection enables different group storage backends
- Variables field stores provider-specific configuration parameters
- Integration with the plugin system in `app/systems/plugins` for provider implementations
- Config mixin provides standardized configuration field management

### Access Control

The group model implements role-based access control as defined in `app/spec/data/group.yml`:
- **Edit permissions**: Available to users with user-admin and config-admin roles
- **View permissions**: Available to users with user-auditor and config-auditor roles

### Base Model Inheritance

The model extends the `name_resource` base model defined in `app/spec/base/data.yml`:
- Uses name as primary key for direct identification
- Provides automatic created/updated timestamp fields
- Implements consistent ordering by name field
- Provides resource mixin functionality

### Mixin Composition

The model incorporates mixins from `app/spec/mixins/data.yml`:
- **ProviderMixin**: Enables plugin-based provider functionality with config and variables fields
- **GroupMixin**: Provides group-based access control through many-to-many relationships
- **ResourceMixin**: Provides core resource functionality and metadata management

### Naming Conventions

- Model files are consistently named `models.py`
- Migration files follow Django's sequential naming pattern
- Class names use PascalCase and are descriptive of their domain (Group, GroupFacade)
- Field names use snake_case following Django conventions
- Database table uses prefixed name (core_group)

### File Organization

Files are organized by Django application conventions:
- Model definition in `models.py`
- Database migrations in the `migrations` subdirectory
- One migration file per schema change

### Domain-Specific Patterns

- The model extends the `name_resource` base model from `app/spec/base/data.yml` for consistent resource behavior
- Uses facade pattern from `app/systems/models` for business logic implementation
- Integrates with Django's migration system for schema version control
- Implements the `provider` mixin for plugin-based functionality
- Uses the `name_resource` base which uses name as primary key
- Implements ordering by name for consistent group listing
- Implements auto_create: true for automatic group creation during system initialization
